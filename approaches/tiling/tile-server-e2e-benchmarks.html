Tile Server End to End Benchmakrs

This notebook parses results from end-to-end testing as documented in tile-benchmarking/e2e/e2e.ipynb.

Setup

Import libraries, define results location and define plotting functions.

import pandas as pd
import numpy as np
import matplotlib.pyplot as plt
                        
git_url_path = "https://raw.githubusercontent.com/developmentseed/tile-benchmarking/main/e2e"

def generate_zooms_plot(df, dataset):
    zooms = [int(path.split('/')[2]) for path in df['Name'][:-1]]
    # Create a new figure
    plt.figure()

    # Create a scatter plot
    plt.scatter(zooms, df['Median Response Time'][:-1])

    # Add title and labels
    plt.title(dataset)
    plt.xlabel('Zoom')
    plt.ylabel('Time (ms)')

    # Display the plot
    plt.show() 
    
def git_url(filename):
    return f'{git_url_path}/{filename}'

def add_to_specs(collection_name: str, df: pd.DataFrame):
    zarr_specs.loc[collection_name, 'mean median response time'] = np.mean(df['Median Response Time'])
    zarr_specs.loc[collection_name, 'median median response time'] = np.median(df['Median Response Time'])

Parse collection results into a data frame

Benchmarks are reported alongside specifications of the dataset, such as chunk shape and size.

zarr_specs = pd.read_csv(f"{git_url_path}/zarr_info.csv")
zarr_specs.index = zarr_specs['collection_name']

for collection_name in zarr_specs.index:
    filename = f'results/{collection_name}_urls_stats.csv'
    results_df = pd.read_csv(git_url(filename))
    add_to_specs(collection_name, results_df)

Review the results

Mean median response time represents the average median response time for all tiles tested. Tested tiles ranged from zoom 0 to zoom 7. From testing we noticed that the most important factors in determining response time were the number of coordinate chunks and the chunk size.

Based on this, the recommendation would be to create chunks as small as possible (e.g. 1 timestep per chunk for the full spatial extent if visualization is the main goal) and not to chunk coordinate data.

However more testing on a high resolution dataset is needed to determine when data should be chunked spatially to deliver good performance. For example, the gpm3imergdl has large chunks because all spatial data is stored in one chunk, but each chunk also has 10 time steps. If each chunk had 1 timestep, each chunk would be ~24.7MB. How would this compare with the performance of the same dataset, chunked spatially into even smaller chunks?
